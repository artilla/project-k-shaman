# 청취 QA 루브릭 — narration v1.1 결정용

목적: `synthesize_tts.py`로 만든 A(LLM narration)·B(서버 조립 narration) 오디오를 들어보고, **LLM 출력에서 narration을 제거하고 서버가 조립해도 음성 품질이 유지되는지** 판정한다. 결과로 `fortune-schema.v1.json`/프롬프트의 v1.1 변경 여부를 결정한다.

## 평가 전 메모

- 비용 측면: 두 버전의 **TTS 오디오 길이·요금은 거의 동일**하다(예측 차 1~2초). 비용 이득은 LLM **출력 토큰 59% 절감**에서 나온다(`docs/reports/token-optimization-report.md`).
- 따라서 이 QA는 "비용 vs 품질"이 아니라 **"품질이 충분히 유지되는가"** 만 본다. 유지되면 B 채택(비용 이득), 부자연스러우면 A 유지 또는 중간안.
- 실질 차이는 **scores 세그먼트 한 곳**에 집중된다(나머지는 동일 텍스트). 거기에 집중해 듣는다.

## 평가 항목 (샘플 5개 × A/B, 각 1~5점)

| 항목 | 5점 기준 | 1점 기준 |
| --- | --- | --- |
| 자연스러움 | 사람이 말하듯 매끄럽다 | 기계적/끊김이 거슬린다 |
| 세그먼트 연결 | 인사→본문→축원이 자연스럽게 이어진다 | 문장 사이가 뚝뚝 끊긴다 |
| scores 문장 | 점수 설명이 자연스럽고 캐릭터 톤에 맞는다 | 템플릿 티가 나고 어색하다 |
| 캐릭터성(홍연) | 밝고 리듬감 있는 홍연 말투가 산다 | 톤이 밋밋/무관하다 |
| 전체 인상 | 그대로 출시 가능 | 수정 없이는 불가 |

## 합격 기준(제안)

- B의 **scores 문장 평균 ≥ 4.0** 그리고 **세그먼트 연결 평균 ≥ 4.0** 이면 → B(서버 조립) 채택, v1.1에서 `narration` 제거.
- 둘 중 하나라도 < 3.5 이면 → **중간안**: LLM이 `scores_line`(짧은 1문장)만 추가로 출력하고 나머지는 서버 조립. (출력 토큰 소폭 증가, scores 자연스러움 보존)
- B 전반이 < 3.0 이면 → A(현재) 유지, narration LLM 출력 존속.

## 추가 점검(기술)

- 예측 길이 vs `out/results.json` 실측 길이 오차 → 단위 경제 TTS 초(28초) 가정 보정.
- 음색 후보 2종(예: `--voice coral`, `--voice sage`) 비교 후 홍연 기본 음색 결정.
- presynth 세그먼트(`--segments`)를 따로 합성해 사전 합성 캐시 품질·이음새 확인.
- iOS Safari 등 실제 단말 재생 테스트(autoplay·세션 중단, §11.2).
