## 1. Office Hours — 6가지 필수 질문

> **AI 초안 — 검토 필요.** 기존 기획 문서(`Plan.md`, `today-shindang-service-plan-v3.md`, `문서-분석-리포트-2026-05-29.md`)에서 추출한 초안입니다. 괄호 안 `(§출처)`로 근거를 표시했으니 사실·범위·지표를 확인하고 다듬으세요.

1. **누가 이것을 원하는가?** (1차 사용자)
   - 운세·사주·타로·K-pop·캐릭터 콘텐츠에 익숙한 **18–35세 모바일 사용자**. 출근·등교·취침 전 1분 안에 기분 전환을 원하는 사람. (v3 §5)
   - 지금은 점신/포스텔러(운세 정보 플랫폼)·헬로우봇(운세 챗봇) 등 **텍스트 중심** 앱으로 해결 중. (v3 §3)
2. **그들이 진짜로 풀고 싶은 문제는?** (Job-to-be-done)
   - 운세의 정보량·정확도가 아니라, **매일 1분짜리 캐릭터 운세 '공연'**을 가볍게 즐기는 것 — 캐릭터 선택 → 음성 청취 → 결과 카드 저장/공유 → 다음 날 재방문. (Plan.md §1, v3 §4)
3. **왜 지금인가? / 왜 우리인가?**
   - 운세 정보 시장은 이미 성숙(사주·타로·챗봇)했지만 **'엔터테인먼트형 캐릭터 음성 공연' 포지션은 비어 있음**. (v3 §3 포지셔닝)
   - LLM·TTS 품질이 저비용 실시간 캐릭터 음성 운세를 가능케 하고, seed·script 캐싱으로 원가를 통제. 차별점은 TTS·캐릭터 모션 자체. (v3 §4, Plan.md §10)
4. **성공의 정의는?** (지표 + 임계값)
   - ① 운세 생성 완료율 **≥ 70%**
   - ② 탭 후 첫 반응(사전합성 인사 + 요약/로딩 UI) **≤ 3초** — 개인화 본문 음성은 별도 지표 (v3 §11.1 재정의)
   - ③ TTS 80%+ 지점까지 **청취 완료율 ≥ 60%**
   - ④ 결과 카드 **저장/공유율 ≥ 10%** (이상 Plan.md §1)
5. **무엇을 만들지 않을 것인가?** (Non-goals)
   - 실시간 상담·실제 상담사 연결·3D VRM·음소 기반 립싱크 **제외**. (Plan.md §2)
   - 베타는 무료 캐릭터 **1종(홍연)**만 — 다캐릭터 동시 선택·커스텀 보이스 계약은 후속(v1.1 A/B). 사주 정밀도 등 **정보량 경쟁도 비목표**. (v3 §7)
6. **N주 안에 측정 가능한 결과는?**
   - Sprint 0–4(**약 9주**) 내 **비공개 베타 배포**, 위 4대 성공지표 + 원가·캐시 적중률 실측. (Plan.md §12, v3 §19)

## 2. 범위 / 아키텍처 개요

> 시스템 경계·주요 컴포넌트·데이터 흐름. (티켓 spec_ref 앵커 대상)

흐름: **PWA**(Next.js/Vite React + TS) → **Fortune API** → Seed Builder → Text/TTS 캐시 → (miss 시) LLM 생성 + Safety Filter → TTS Provider Adapter → Object Storage/CDN → Avatar Playback → Result Card.

## 3. 권한 경계 (가역성 기준)

> 자동 허용(loop) vs 인간 승인(hold). 판단 기준: "잘못됐을 때 즉시 되돌릴 수 있는가?"
> 상세는 docs/runbook.md §4 참조.

- **자동 허용 후보**: 프롬프트/스키마 문서화, mock 데이터·테스트 추가, `narration_composer.py` 등 순수 로직 리팩토링, 분석 이벤트 taxonomy 문서.
- **인간 승인 필수**: LLM/안전 필터 정책 변경, 개인정보(생년월일/출생시간) 처리·HMAC 키, 과금/PG, 실제 TTS 비용 발생 호출, 운영 배포. (개인정보·결제·외부 비용은 가역성 없음)
