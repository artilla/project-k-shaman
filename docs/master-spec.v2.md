## 1. Office Hours — 6가지 필수 질문

1. **누가 이것을 원하는가?** (1차 사용자)
   - K-콘텐츠/K-pop 무대 감성에 익숙하고 운세를 가볍게(오락+자기성찰) 소비하는 모바일 사용자.
     모바일 웹/PWA 접근, 매일 1분 내외의 짧은 세션. (v3 §1–§2)
   - "정확한 사주풀이"가 아니라 **캐릭터가 나에게 말을 걸어주는 경험**을 원하는 라이트 유저.

2. **그들이 진짜로 풀고 싶은 문제는?** (Job-to-be-done)
   - "오늘 하루를 시작할 기분 전환 리추얼이 필요하다 — 나만을 위한 짧은 공연처럼."
   - 기존 운세 앱은 정보 플랫폼(점신·포스텔러)이거나 챗봇(헬로우봇)이라 **무대형 시청각
     경험(음성+모션+공유)의 공백**이 있다. 오늘신당의 핵심 경험은 "운세를 읽는 앱"이 아니라
     **"나만을 위한 1분짜리 굿/무대 퍼포먼스"** (v3 §1, §3 포지셔닝).

3. **왜 지금인가? / 왜 우리인가?**
   - 지금: TTS·LLM 단가 하락으로 개인화 음성 콘텐츠의 단위 경제가 성립 구간에 진입 (v3 §13).
     2026-06-13 리서치 기준 동일 포지션 서비스 미확인 — 공백 유효. 최우선 잠재 추격자(헬로우봇의
     음성 운세)가 움직이기 전 선점 (v3 §3.3 모니터링 트리거).
   - 우리: 엔진 자산 확보 완료 — 스키마 v1.1+검증기(T001), 홍연 캐릭터 시트(T002), narration
     조립 확정(T003, "scores_line 중간안"), TTS 프로바이더·음색 선정(T005/ADR-0001), 공유 카드
     (T006), mock 기반 API·seed·TTS·캐시 통합(T010–T016), 단위경제 시뮬레이터.

4. **성공의 정의는?** (지표 + 임계값)
   - KPI (v3 §17): D1/D7/D30 재방문율 · 운세 생성 완료율 · TTS 첫 재생 시간 · 청취 완료율 ·
     공유율 · TTS 비용/사용자 · 캐시 히트율.
   - 확정 임계값: **첫 화면 체감 3초 재생 경로**(사전합성 세그먼트 우선 + 텍스트 먼저, v3 §10–11) ·
     **cache miss 신규합성 기여분 ≤ $0.01/세션** (ADR-0001, v3 §13).
   - 베타 리텐션·전환 임계값: `[운영자 확정]` (v3 §13 사내 실측 단가·§22 체크리스트 입력 후).

5. **무엇을 만들지 않을 것인가?** (Non-goals, 베타 기준)
   - 캐릭터 선택 2종 — 베타는 **홍연 1종 고정**, 선택 재미는 v1.1 A/B (v3 §7).
   - 커스텀 보이스 계약(기본 음색만), 네이티브 앱(모바일 웹/PWA만), 상담·과금 고도화.
   - streak/친밀도/푸시/주간 리포트 — 후속으로 분리 (v3 §9·§15).
   - 운세 "정확도" 경쟁 — 콘텐츠는 오락·자기성찰 도구 (v3 §2 전제).

6. **N주 안에 측정 가능한 결과는?**
   - 엔진은 mock 통합(T016)까지 완료. 다음 마일스톤:
     ① presynth+캐시 **실경로** 연결(mock → 실 프로바이더, ADR-0001 어댑터),
     ② 3초 재생 경로 실측 — `tts_generate_start/complete`·`cache_hit/miss` 이벤트 (v3 §17),
     ③ 홍연 운세 1분 공연 end-to-end 베타 데모(모바일 웹, 첫 재생은 사용자 탭 — v3 §11).
   - 기한: `[운영자 확정 — N주]`

## 2. 범위 / 아키텍처 개요

> 시스템 경계·주요 컴포넌트·데이터 흐름. (티켓 spec_ref 앵커 대상)

**데이터 흐름 (베타)**

```
사용자 입력(생년월일 등)
  → seed_builder (T011)
  → fortune API (LLM · 베타는 mock: fortune_api_mock, T010)
  → fortune-schema v1.1 검증 (validate_fortune, T001)
  → narration composer (T003 — greeting→summary→scores_line→…→ending, "scores_line 중간안" 확정)
  → TTS adapter (T013 · 프로바이더/음색 = ADR-0001)
  → cache layer (T015 — presynth 우선, 키는 HMAC 해시만)
  → 재생 (모바일 웹/PWA · 첫 재생은 사용자 탭으로 오디오 컨텍스트 오픈, v3 §11)
  + share card (T006)
```

**컴포넌트 ↔ 테스트 매핑**: `fortune-engine/` — `seed_builder.py`·`fortune_api_mock.py`·
`validate_fortune.py`·`tts_adapter.py`·`cache_layer.py`·`share_card.py`, 각
`tests/test_*.py` (T008 엔진 smoke 포함). 통합: T012(mock+seed) → T014(api+tts) → T016(api+cache).

**개인정보/데이터 경계** (v3 §12·§18): 생년월일·최근 선택 기록은 **서버 저장(동의 기반)** 단일
경로. 캐시 키는 **HMAC 해시만** 사용 — 원문 개인정보가 캐시 계층에 닿지 않는다.

**운영 하네스**: Ralph Loop(명세→티켓→헤드리스→검증→승인) + 프로젝트 전용 Mission Control
(`./ralph/scripts/mission_control.sh start`). 검증 GREEN 기준은 `ralph/scripts/run_checks.local.sh`.

## 3. 권한 경계 (가역성 기준)

> 자동 허용(loop) vs 인간 승인(hold). 판단 기준: "잘못됐을 때 즉시 되돌릴 수 있는가?"
> 상세는 ralph/docs/runbook.md §4 참조.

- **자동 허용(loop)**: `fortune-engine/` 코드·mock·테스트, 문서/티켓 초안, 하네스 스크립트 보수.
  (전부 git revert 가능)
- **인간 승인(hold)**: 실 프로바이더 API 키·과금 호출(TTS/LLM 계약 전환), 개인정보 스키마·저장
  경로 변경(§2 데이터 경계), 가격·단위경제 가정 변경(v3 §13), 외부 배포/공개.
