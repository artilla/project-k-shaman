## 0. 한 줄 요약

이 폴더에는 **두 갈래의 작업물**이 섞여 있다. (1) **오늘신당** — K-pop 판타지 무당 캐릭터 운세 PWA 서비스 기획 및 운세 엔진 실측(주력, 문서·코드 대부분), (2) **에이전트/ADK 리서치 PDF 3종** — 별도 프로젝트(LegalDocs Wise Pro)와 "Agentic Era" 참고 자료로 보이는 부속물. 오늘신당 기획은 v1→v2→v3로 진화하며 "사업 아이디어"에서 **"착수 가능한 실측 기반 문서"**로 성숙했고, 원가(LLM·TTS) 통제가 모든 기술 의사결정의 척추다. 문서 간 수치는 대체로 정합적이며 시뮬레이터 수식까지 검증된다. 다만 일부 리포트의 원가 수치가 버전 시차로 stale 상태이고, 모든 단위 경제는 아직 **추정치(프록시 토큰·공개가·단일 음색)**라는 점이 핵심 한계다.

## 1. 프로젝트 정체성: 오늘신당

`오늘신당`은 한국 무속 모티프 + K-pop 무대 감성을 결합한 모바일 웹/PWA 운세 서비스다. 핵심 경험은 "운세를 읽는 앱"이 아니라 **"나만을 위한 1분짜리 캐릭터 운세 공연을 보는 앱"**으로 정의되어 있다.

- **슬로건**: "오늘의 기운을 무대 위에서 듣다."
- **포지셔닝**: 점신/포스텔러(운세 정보 플랫폼)·헬로우봇(운세 챗봇)과 달리 "매일 1분짜리 캐릭터 운세 공연".
- **타깃**: 운세·사주·타로·K-pop·캐릭터 콘텐츠에 익숙한 18–35세, 출근/등교/취침 전 1분 사용자.
- **차별점**: AI 무당이 움직이고 말하는 음성·모션·팬덤·공유 카드.
- **핵심 리스크(자인)**: 원가 구조와 콘텐츠 품질 관리. → 문서 전체가 이 리스크 방어에 집중.

사용자 흐름은 *무당 선택 → 생년월일/주제 입력 → 등장 연출 → (텍스트 먼저 노출) 운세 음성 재생 → 결과 카드 → 공유/재방문 유도*의 8단계로 일관된다.

## 3. 버전 진화: v1 → v2 → v3 (무엇이, 왜 바뀌었나)

```
v1 (05-21)          v2 (05-22)               v3 (05-22, 현행)
사업 아이디어    →   비용/리스크 방어 보강   →   착수 가능 문서(결정값 확정)
```

**v1 → v2 핵심 추가**
- 단위 경제 가설(손익 식), 5계층 캐싱 전략, 리텐션 루프(streak/친밀도/푸시/리포트), 컴플라이언스(개인정보·미성년자·결제·합성음), 리스크 등록부, A/B 후보.
- 캐릭터 **3종 동시 출시 → 1–2종으로 축소**(제작·QA 원가 리스크).

**v2 → v3 핵심 결정(미결 항목을 값으로 확정)**
- **3초 SLA 재정의**: "완성 음성까지"가 아니라 **"탭 후 첫 반응(사전합성 인사 + 요약 UI)까지"**. 개인화 텍스트/본문 음성은 LLM·TTS 완료에 종속되는 별도 지표로 분리.
- **개인정보 경로 단일화**: 동의 기반 **서버 저장** 확정 + 캐시 키는 **HMAC 해시만**(원본 생년월일 미포함).
- **단위 경제 숫자표**: 공개가 + 실측 기반 1k/10k/50k DAU 손익·break-even 표.
- **재방문 장치 분리**: 핵심 흐름의 친밀도/알림은 "비전"이고 **MVP 실구현은 streak 1종뿐**임을 명문화.
- **모바일 autoplay**: A/B 후보에서 빼고 **자동재생 금지 전제**로 격상(탭으로 AudioContext 오픈).
- **베타 캐릭터 = 홍연 1종 고정**(2종 선택 재미는 v1.1 A/B).
- **실패/지연 이벤트 보강**: `fortune_fail`, `tts_generate_*`, `tts_play_error`, `cache_hit/miss`, `share_fail`.

`docs/planning/Plan.md`는 원래 v2를 기준으로 작성된 **실행 계획**(스프린트·API·DB 모델·QA)이었으나, 2026-05-29에 v3 최신 결정(베타 1종 확정, `scores_line`, 서버 저장 단일화, 신규 이벤트)을 반영해 동기화했다.

## 4. 운세 엔진의 핵심 의사결정 체인 (가장 중요한 분석)

이 프로젝트의 기술적 정수는 **"narration을 누가 만들 것인가"** 결정이며, 측정 → 비용분석 → A/B → 청취QA → 스키마 개정으로 이어지는 추적 가능한 체인이다.

1. **측정**(`fortune-measurement-report`): 샘플 15개 기준 입력 ~2,244 / 출력 ~580 토큰(프록시), 본문 202음절, 전체 TTS 48.2초(45–60초 목표 충족), **캐시 가능 비율 45.3%**.
2. **토큰 최적화**(`token-optimization-report`): ① 컨텍스트 캐시가 즉효 레버(LLM 비용/건 $0.00213→$0.0015, 27%↓). ② 캐시를 켜면 few-shot 트리밍의 추가 이득은 미미. ③ 다음 레버는 **출력 토큰** — `narration`을 LLM이 안 만들고 서버가 조립하면 580→239(59%↓).
3. **A/B 설계**(`ab-comparison`): A·B는 텍스트 대부분 동일하고 **실질 차이는 `scores` 세그먼트 한 문장**뿐. TTS 길이·요금은 사실상 동일 → 이득은 LLM 출력 토큰뿐. 그래서 "비용 vs 품질"이 아니라 **"품질이 유지되는가"**만 보는 QA로 좁힘.
4. **실제 합성**(`tts-ab-results-report`, `coral` 음색 5개): A full 평균 42.1초, 신규합성 31.0초 → 시뮬레이터 TTS 기본값 **28→31초 보정**.
5. **청취 QA 결정**(`listening-decision-report`): 사전 정의한 분기(scores·연결 둘 다 ≥4.0=전면채택 / 하나라도 <3.5=중간안 / 전반 <3.0=현행유지)에서 **B scores 3.0·연결 3.0** → **`scores_line` 중간안 채택**.
6. **스키마 개정**(v1→v1.1): `narration` 제거, `scores_line` 추가, 샘플 15개 마이그레이션, `narration_composer.py`로 서버가 `greeting→summary→scores_line→advice→lucky→avoid→blessing→ending` 조립.

**결과**: LLM은 구조화 필드 + `scores_line`만 출력(출력 토큰 580→**278** 잠정), 서버가 presynth·템플릿으로 TTS 대본 조립. scores 문장의 자연스러움(LLM)과 비용 절감(서버 조립)의 균형점.

## 5. 단위 경제 — 검증 결과

`unit-economics-simulator.xlsx`(시트 "시뮬레이터")의 수식을 직접 재계산했고, **v3 §13의 모든 수치가 시뮬레이터에서 정확히 도출됨을 확인**했다.

**건당 원가(컨텍스트 캐시 ON, v1.1 출력 278토큰 기준)**
- LLM/건 = 캐시읽기(2,164×$0.03/1M) + 가변입력(104×$0.30/1M) + 출력(278×$2.50/1M) = **$0.000791** ✓
- TTS/건(miss) = 31초/60 × $0.015 = **$0.00775** ✓
- 인프라/건 = $0.0005
- **cache MISS 원가/건 ≈ $0.0090** ✓ / **cache HIT 원가/건 = $0.0005** ✓
- 컨텍스트 캐시 스토리지 ≈ $0.052/일(요청에 분산되어 무시 가능) ✓

**DAU별 월 무료 변동비 (재계산 일치)**

| DAU | 베타 캐시 30% ($0.00648/일) | 정식 캐시 60% ($0.00392/일) |
| --- | --- | --- |
| 1,000 | ~$194/월 | ~$117/월 |
| 10,000 | ~$1,943/월 | ~$1,175/월 |
| 50,000 | ~$9,717/월 | ~$5,874/월 |

**Break-even(변동비 기준)**: 유료 ARPU net = ₩3,900×0.9÷1,380 = **$2.54**. 정식 60%·10k DAU(MAU 30k) 기준 월 변동비 $1,175 ÷ $2.54 ≈ **462명** = **MAU의 1.54%** 전환 시 변동비 손익분기. ✓ (모두 시뮬레이터와 일치)

**청취 점수 평균(원본 JSON 재집계, 리포트와 일치)**: A 자연스러움 2.8·연결 3.2·scores 3.0·캐릭터 3.0·전체 3.0 / B 3.2·3.0·3.0·3.0·3.6.

## 6. 일관성 점검 — 발견된 불일치/주의점

수치는 전반적으로 정합적이나, 버전 시차로 인한 **stale 수치/주석 4건**을 확인했다. 모두 의사결정을 바꾸진 않지만 정리해 두는 것이 좋다.

> **반영 현황(2026-05-29)**: 1번 → `tts-ab-results-report.md` §3에 정정 주석 추가(정본 $0.0090 명시). 2번 → xlsx A2/A54 주석 정정(baseline이 이미 v1.1 278tok임을 반영), LibreOffice 재계산으로 수식 오류 0 확인. 3번 → 변천 사슬은 v3가 통합(정본 $0.0090). 4번 → 표본 한계는 기록만 유지(결정은 분기 기준상 견고). 추가로 `docs/planning/Plan.md`를 v3 결정값으로 동기화 완료.
